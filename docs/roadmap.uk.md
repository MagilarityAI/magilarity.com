### td_creator — Конструктор тендерної документації для замовників (спроектовано 10.07.2026)
Перший агент для ЗАМОВНИКА: вид закупівлі → базові вимоги категорії (авто-галочки за
частотою) → специфічні вимоги з ЖИВИМИ ризик-попередженнями (прецеденти реальних рішень
ООПЗ + безпечні формулювання) → параметри з медіанами/безпечними діапазонами → проєкт ТД
(Word) + супровідна записка. Нормативний каркас — затверджена примірна форма ТД + профільні
НПА категорії. Два режими: з нуля / від чернетки замовника. Self-check згенерованої ТД
власним аналізатором (tender_doc_researcher) — «очима учасника і скаржника» до публікації.

**База вимог:** наповнюється з масової обробки реальних ТД (полегшений extraction-режим
Gemini) + як побічний продукт кожного прогону tender_doc_researcher; вимоги зберігаються
канонізовано (частоти, параметри, ризик-профілі з бази рішень ООПЗ). UI — Streamlit MVP.

### investigation — Оркестратор розслідування змов
**Статус:** архітектура v2 спроектована 07.07.2026, звірена повузлово зі схемою drawio
(Фаза 0 скоупінг-checkpoint, ранні вимоги, перевірка ЄДР перенесена у Фазу 1). Код ще не
почато. Залежить від наповнення таблиці знань `amku_researcher` (джерело few-shot для
синтезу) та таблиці прецедентів `oopz_researcher` (обидві структурно готові після ремонту
06–10.07.2026).

Архітектура v2 слідує офіційному алгоритму спеціаліста АМКУ (повузлово звірена зі схемою
доведення) і працює у трьох фазах:

| Фаза | Зміст |
|------|-------|
| Фаза 0 — Скоупінг | коло торгів суб'єктів → таблиці → checkpoint: спеціаліст звужує предмет розслідування |
| Фаза 1 — Prozorro-аналіз | матриці спільних ознак учасників: властивості файлів, час завантаження, однакові/ідентичні документи, контакти, спільні працівники/МТБ, синхронність дій, цінові пропозиції + генерація пакета офіційних вимог (ДПС/ПФУ/банки/майданчики) |
| Фаза 2 — Відповіді органів | інжест офіційних відповідей (IP-адреси, e-mail, фінансові зв'язки) → фінальний синтез |

**Висновок — 10 обов'язкових пунктів** (кожен: наявність АБО обґрунтована відсутність
ознаки; повнота контролюється детерміновано, жоден пункт не може бути пропущений мовчки) →
проект подання АМКУ. Кожен доказ — verbatim-цитата з точним джерелом. Скоуп MVP —
виключно bid rigging (п.4 ч.2 ст.6 ЗУ «Про захист економічної конкуренції»).

### bid_creator — Генератор пакета документів учасника (спроектовано 10.07.2026)
Восьмий агент, дзеркальний до td_creator: за чеклістом вимог від tender_doc_researcher
генерує пакет документів пропозиції. Прості документи (довідки, гарантійні листи, згоди) —
повністю, з дослівною прив'язкою кожного до вимоги ТД; складні (техспецифікації, банківська
гарантія) — чесно «відповідальність користувача» з максимальною допомогою (заявка до банку
з параметрами з ТД, чек-листи отримання). Профіль організації зі сховищем постійних
документів і контролем строків придатності; стильова варіативність + обов'язковий
дисклеймер; фазові пакети за хронологією тендера (24-год виправлення за звітом
bid_researcher, документи переможця з дедлайнами, підписання). Незалежний self-check
згенерованого пакета контролером bid_researcher. Мультитенантність для зовнішніх
користувачів. UI — Streamlit MVP.

## 🚀 Фаза 2 — Масштабування на ринку України

### Розширення функціональності платформи

**Повний customer_profiler — Профіль замовника**
- Рейтинг репутації замовника на основі рішень ООПЗ, судових справ і статистики закупівель
- Виявлення патернів зловживань конкретного замовника
- Прогнозування поведінки замовника для нових тендерів
- Категоризація: trusted / caution / high_risk

**Агент розгляду скарг ООПЗ** — див. `complaint_researcher` у Фазі 1 вище (спроектований
07.07.2026; тут — його розгортання до повноцінного інструмента колегії після наповнення
бази прецедентів).

**Автоматична подача скарг до ООПЗ**
- Генерація повного пакету скарги на основі аналізу tender_doc_researcher
- Автоматичне заповнення форм
- Відстеження статусу розгляду

**Платформа: кабінети користувачів + бекенд (СПРОЕКТОВАНО 11-12.07.2026)**
- Два типи користувачів: УЧАСНИК (кабінет: вхід з номера закупівлі, фазовий шлях тендера —
  аналіз → пакет → 24-год → результат → договір) і ЗАМОВНИК (конструктор ТД: вхід з виду
  закупівлі, покроковий майстер) — концепції всіх екранів + живі HTML-мокапи готові
- Спільний бекенд: FastAPI + черга задач для довгих аналізів + auth + білінг
  (підписки з квотами + метровані преміум-дії) + Telegram-сповіщення (дедлайни,
  обов'язкові дії замовника, «результат готовий»)
- Стратегія: все на Streamlit спочатку → повноцінний фронтенд через ті самі API
- Внутрішні інструменти: операторська консоль баз знань (рев'ю-стіл, batch-обробка
  з бюджет-контролем), модуль новин і тижневої статистики оскаржень з детектором
  зсуву практики ООПЗ

## 🌍 Фаза 4 — Вихід на ринок Європи

### Чому логіка Magilarity переноситься в ЄС

```
Єдина директива ЄС 2014/24 про публічні закупівлі
діє в усіх 27 країнах-членах ЄС

Спільна логіка для всіх:
├── Публічні закупівлі через єдині платформи
├── Антиконкурентні узгоджені дії (Art. 102 TFEU)
├── Органи оскарження (аналог ООПЗ)
├── Антимонопольні комітети (аналог АМКУ)
└── Відкриті дані закупівель

= Архітектура агентів переноситься
  змінюється тільки мова і локальне законодавство
```

### Пріоритетні ринки

**🇵🇱 Польща — перший ринок ЄС**
- Найбільший ринок публічних закупівель в ЦСЄ (~€50 млрд/рік)
- Схожі проблеми з корупцією в закупівлях
- Велика українська діаспора як перші клієнти
- Платформа: Platforma e-Zamówienia (аналог ProZorro)
- Орган оскарження: KIO (аналог ООПЗ)
- Строки локалізації: 1-2 місяці

**🇷🇴 Румунія**
- Активний ринок закупівель (~€15 млрд/рік)
- Значні проблеми з прозорістю
- Платформа: SEAP
- Висока зацікавленість антикорупційних організацій

**🇧🇬 Болгарія**
- Схожі системні проблеми із закупівлями
- Активна громадянська антикорупційна спільнота

**🇩🇪 Німеччина — довгострокова ціль**
- Найбільший ринок закупівель ЄС (~€500 млрд/рік)
- Платформа: DTVP, Vergabe24
- Вимагає повної локалізації та партнера

### Стратегія виходу в ЄС

```
КРОК 1 — Технічна підготовка:
├── Мовна локалізація (польська)
├── Завантаження бази польського законодавства
└── Підключення польської платформи закупівель API

КРОК 2 — Партнерство:
├── Юридична фірма-партнер в Польщі

КРОК 3 — Пілот:
├── 10-20 польських компаній-учасників закупівель
├── Верифікація якості висновків місцевим юристом
└── Адаптація промтів під польську правову практику

КРОК 4 — Масштабування:
└── Румунія → Болгарія → інші країни ЄС
```

## 💎 Стратегічне бачення

> *Magilarity — стати стандартом де-факто для аналізу публічних закупівель в Центральній та Східній Європі. Як YouControl став стандартом перевірки контрагентів в Україні — Magilarity стане стандартом для роботи з держзакупівлями.*

**Довгострокова мета:** Інфраструктурна платформа що обслуговує всіх учасників екосистеми державних закупівель — від малого бізнесу що вперше бере участь у тендері до антикорупційних органів що розслідують системні зловживання.
