docker-compose up --build

## 🗄️ Бази даних

### APPDB (appdb) — Тільки читання

Законодавча база даних для української юрисдикції.

| Вміст | Масштаб |
|-------|---------|
| Повний текст українського законодавства | 286,634 нормативних документів |
| Рішення ООПЗ | 4,400+ рішень |
| Рішення АМКУ | Проіндексовано |
| Судові рішення | Таблиця готова, наповнення заплановано |
| Статті Закону №922-VIII | Повний текст |
| Постанова КМУ №1178 | Повний текст |

**54 таблиці.** Агенти мають доступ тільки для читання. Всі юридичні перехресні посилання та запити валідації виконуються проти цієї бази.

**Автоматичне оновлення:** щоденна синхронізація з data.gov.ua API — нові нормативні документи завантажуються автоматично.

## 📄 Pipeline обробки документів

```
Вхід: PDF / DOCX / XLSX / архіви
    ↓
Розпаковка архівів (.zip / .rar / .7z)
    ↓
Конвертація Office: DOCX→PDF, XLSX→CSV (openpyxl)
    ↓
Витяг тексту: python-docx (основний), pdfplumber, openpyxl
    ↓
PDF → Markdown: конвертація з LLM корекцією артефактів
(економія 50-70% токенів порівняно з сирим PDF)
    ↓
OCR fallback: Tesseract (для сканованих PDF)
    ↓
PDF→PNG: PyMuPDF/fitz (для vision LLM викликів)
    ↓
LLM аналіз (гібридна маршрутизація моделей)
    ↓
Вихід: JSON + DOCX звіти
```

## 🔐 Безпека користувачів та захист системи

### Безпека доступу користувачів

**Автентифікація та авторизація:**
- JWT токени з коротким терміном дії (access token 15 хв, refresh token 7 днів)
- Двофакторна автентифікація (2FA) для всіх акаунтів
- Role-Based Access Control (RBAC) — розмежування прав між ролями:
  - `participant` — учасник закупівель (аналіз ТД, перевірка пропозицій)
  - `buyer` — замовник (генератор ТД, аналіз пропозицій учасників)
  - `lawyer` — юридична фірма (розширений доступ, API)
  - `admin` — адміністратор платформи
- Захист від брутфорсу — блокування після 5 невдалих спроб входу
- Сесії з автоматичним завершенням при неактивності

**Захист даних користувачів:**
- Паролі зберігаються виключно у вигляді bcrypt хешів
- Персональні дані шифруються at-rest (AES-256)
- Передача даних тільки через HTTPS/TLS 1.3
- Розмежування даних між клієнтами (tenant isolation)

---

### Безпека API та запитів

**Rate limiting:**
- Обмеження кількості запитів на користувача (per-user rate limiting)
- Окремі ліміти для різних типів операцій:
  - Аналіз ТД: не більше 50 запитів/день на базовому плані
  - API виклики: ліміти по токенах залежно від тарифу
- Захист від DDoS через Cloudflare

**Валідація вхідних даних:**
- Перевірка всіх вхідних параметрів (номер тендеру, ЄДРПОУ тощо)
- Захист від SQL ін'єкцій (параметризовані запити)
- Санітизація завантажуваних файлів
- Обмеження розміру файлів що завантажуються

### Захист інфраструктури

**Мережева безпека:**
- Всі компоненти в приватній мережі (VPC)
- Доступ до БД тільки з внутрішньої мережі (без публічного IP)
- Firewall правила — відкриті тільки необхідні порти
- VPN для адміністративного доступу

**Безпека контейнерів:**
- Docker образи на базі мінімальних base images (alpine)
- Регулярне оновлення залежностей і патчів безпеки
- Сканування образів на вразливості (Trivy / Snyk)
- Контейнери запускаються від non-root користувача
- Read-only файлова система де можливо

**Секрети та конфігурація:**
- API ключі та паролі зберігаються у `.env` файлах (не в коді)
- У продакшн середовищі — Google Secret Manager або HashiCorp Vault
- Ротація API ключів за розкладом
- `.gitignore` — всі секрети виключені з репозиторію

### Безпека баз даних

**Права доступу (принцип мінімальних привілеїв):**
- APPDB (законодавство) — тільки `SELECT` для всіх агентів
- agent_memory — `SELECT/INSERT/UPDATE` тільки для відповідних агентів
- prozorro_intel — розмежований доступ між агентами
- Окремий користувач БД для кожного сервісу

**Резервне копіювання:**
- Автоматичні щоденні backup всіх баз даних
- Зберігання backup в окремому geographic регіоні
- Перевірка відновлення backup щомісяця
- Point-in-time recovery для критичних БД

## 🔮 Запланована інфраструктура

### Векторна БД — семантичний пошук

Перехід від SQL до гібридного підходу SQL + векторний пошук:

```
ПОТОЧНИЙ ПІДХІД (SQL):
SELECT * FROM oopz_decisions
WHERE dk_code LIKE '45%'
→ Точний пошук за кодом ДК

МАЙБУТНІЙ ПІДХІД (Vector + SQL):
Семантичний запит: "дискримінаційні вимоги
до досвіду в будівельних тендерах"
→ Знаходить релевантні рішення ООПЗ
  навіть якщо точні слова не співпадають
→ Набагато вища якість прецедентів
```

**Планові векторні бази:**
- `pgvector` (розширення PostgreSQL) — без окремого сервера
- Або `Qdrant` / `Weaviate` при масштабуванні
- Embeddings для: рішень ООПЗ, статей законодавства, еталонних аналізів

**Коли переходити:** після накопичення 10,000+ проаналізованих рішень ООПЗ — тоді семантичний пошук дасть суттєву перевагу над точним SQL.

### Масштабування інфраструктури

**Поточний стан:** монолітний Docker на одному сервері

**Наступний етап (при зростанні навантаження):**

```
Load Balancer
    ↓
┌─────────────────────────────────┐
│ API Gateway (FastAPI)           │
└──────────┬──────────────────────┘
           ↓
┌──────────────────────────────────────┐
│ Черга задач (Celery + Redis)         │
│ Асинхронна обробка тендерів          │
└──────┬───────────────────────────────┘
       ↓
┌──────────────────────────────────┐
│ Пул агентів (Docker Swarm/K8s)  │
│ Горизонтальне масштабування      │
└──────┬───────────────────────────┘
       ↓
┌──────────────────────────────────┐
│ PostgreSQL кластер               │
│ Read replicas для APPDB          │
└──────────────────────────────────┘
```

**Хмарна інфраструктура (Google Cloud):**
- Cloud Run — автомасштабування агентів
- Cloud SQL — керований PostgreSQL
- Vertex AI — уніфікований доступ до AI моделей
- Cloud Storage — зберігання документів і звітів
- BigQuery — аналітика по 50,000+ учасниках ProZorro
