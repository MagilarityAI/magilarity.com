# 🖥️ Архітектура платформи (фронтенд + бекенд)
**Статус:** Спроектовано (липень 2026), реалізація попереду — архітектура бекенда 11.07.2026, концепції інтерфейсів 11–12.07.2026
**Роль:** Продуктовий шар, що перетворює агентів на сервіс: кабінети користувачів для обох сторін ринку закупівель, API gateway, черга задач, білінг і сповіщення

---

## Два користувачі, дві точки входу

Платформа обслуговує обидві сторони ринку закупівель, кожна зі своїм кабінетом і своєю природною точкою входу:

| Користувач | Кабінет | Все починається з… |
|------------|---------|---------------------|
| **Учасник** (директор МСБ / тендерний менеджер / юрист, що веде кількох клієнтів) | Кабінет учасника | **номера закупівлі Prozorro** — кожна функція (звіт, чеклист, пакет, аналіз переможця, скарга) — гілка від конкретного тендера |
| **Замовник** (уповноважена особа установи, часто без юриста) | Кабінет-конструктор ТД | **виду закупівлі** — вибір категорії або вільний ввід назви предмета / коду ДК з автоматичною класифікацією |

---

## Флоу учасника: фазовий шлях, а не вкладки

Деталка тендера слідує реальному життєвому циклу тендера — інтерфейс «веде за руку»; майбутні кроки видимі, але замкнені до відкриття своєї фази:

```
КРОК 1. РІШЕННЯ   («заходити чи ні?»)  → аналіз ТД · експертиза договору · профіль замовника ·
                                          перевірка заточеності · [оскаржити умови ТД до дедлайну]
КРОК 2. ПІДГОТОВКА                     → чеклист · генерація пакета · перевірка ВЛАСНОГО
                                          пакета до подання · інструкція подання
КРОК 3. РОЗГЛЯД                        → [якщо відкрилось 24-год вікно виправлення: тривога + регенерація]
КРОК 4. РЕЗУЛЬТАТ                      → перемога: winner-пакет з 4-денними дедлайнами
                                          програш: аналіз пропозиції ПЕРЕМОЖЦЯ → підстави оскаржити (вікно 4 дні)
КРОК 5. ДОГОВІР                        → signing-пакет · фінальні нагадування
```

Наскрізно: повна хронологія віх і дедлайнів та майстер скарги, що живиться підставами з аналізів. Дедлайни розраховуються з дат Prozorro і норм права, ніколи не хардкодяться.

## Флоу замовника: майстер, а не форма

Створення ТД — покроковий wizard (одне рішення на крок, з поверненням назад): вид закупівлі → реквізити → базовий пакет вимог (галочки ввімкнені) → специфічні вимоги з живими прецедентними ризик-попередженнями → зведене ризик-рев'ю → генерація → опційний self-check «очима учасника і скаржника». Кабінет замовника також веде його по власних законних обов'язках (строки розгляду пропозицій, обов'язок 24-годинного виправлення, таймери оприлюднень) з нагадуваннями.

## Фронтенд

- **Дизайн-система — «темний chrome + світле полотно».** Навігація, сайдбар і футер несуть темну бренд-мову (продовження публічного сайту); робоче полотно — звіти, чеклісти, конструктори — світле («папір»): користувачі годинами читають щільний юридичний текст, а полотно відповідає білим DOCX-результатам, які вони завантажують.
- **Єдина статусна мова** для всіх продуктів (готово / у роботі / спроектовано / порушення / бейджі уваги «вирішує користувач»), з правилом доступності: статус ніколи не передається лише кольором.
- **Поетапна доставка:** спершу Streamlit MVP; усе, що бачить користувач, ходить до бекенда **через API з першого дня** — тому пізніший повний фронтенд замінює лише UI-шар: той самий API-контракт, нуль змін бекенда.
- **Telegram-сповіщення** як повноцінний канал поряд з бейджами кабінету та e-mail. Кожен довгий аналіз завершується push-повідомленням «✅ Готово — Відкрити», тому прогрес-екрани чесно кажуть «можете закрити сторінку — повідомимо, щойно буде готово». Термінові класи дедлайнів (24-годинне вікно виправлення, дедлайн оскарження завтра) завжди йдуть у Telegram. Кожне сповіщення — одна дія: терміновість + що зробити + до якої дати + одна кнопка-deep-link.

## Бекенд

Місія бекенда: зробити агентів продуктом, не дублюючи їх. Агенти вже будуються як чисті Python core-бібліотеки — бекенд є **тонкою обгорткою над цими core плюс інфраструктурними підсистемами**. Він не містить бізнес-логіки агентів, не рендерить UI і не зберігає знання.

| Підсистема | Призначення |
|------------|-------------|
| **API Gateway (FastAPI)** | Домени маршрутів дзеркалять екрани кабінетів; версіонована OpenAPI-схема — обов'язковий контракт для кожного покоління фронтенду |
| **Черга задач** | Серце бекенда — аналізи тривають хвилини й години. Повний життєвий цикл задачі зі статусами і прогресом; етапи прогресу читаються з checkpoint-логів самих агентів; ретраї транзієнтних помилок; ідемпотентні перезапуски продовжують з checkpoint (агенти вже вміють resume). Перед кожним платним запуском — кошторис |
| **Auth + tenancy** | Ролі (учасник / замовник / staff); ізоляція клієнтів забезпечується у кожному шарі доступу до даних, включно з файлами; прив'язка Telegram одноразовим кодом |
| **Білінг** | Тарифи-підписки з квотами на регулярні дії + метровані доплати за епізодичні LLM-важкі фічі; єдиний резолвер платної дії, щоб UI завжди показував стан (включено / доплата / апгрейд) *до* кліку; **cost ledger** фіксує і ціну для клієнта, і LLM-собівартість по кожній задачі для юніт-економіки; карткова оплата — лише через hosted-сторінку платіжного провайдера |
| **Сервіс сповіщень** | Канали: бейдж кабінету, e-mail, Telegram-бот; класи сповіщень з правилами тихих годин; «готовність результату» доставляється завжди |
| **Планувальник** | Cron-джоби розраховують фазові віхи тендерів з дат Prozorro (дедлайни оскарження, 24-годинні вікна, строки переможця, обов'язки замовника) і живлять і сповіщення, і фазові стрічки UI — одна модель даних для обох |
| **Адаптери агентів** | По одному на агента, єдиний контракт: кошторис → запуск → прогрес → результат. Прямі виклики core-бібліотек, без subprocess |

Продуктові дані живуть в окремій прикладній базі даних, свідомо відділеній від сховищ знань агентів; існуючі бази зберігають свої чинні правила доступу.

## ☁️ План використання хмарних кредитів

Бази знань платформи — її головна перевага, а їх наповнення — фундаментально **задача LLM-об'ємів**. Агенти вже працюють на Gemini (2.5 Flash для аналізу та екстракції, 3.5 Flash там, де він доповнює — включно з ансамблем двох моделей для порівняння техспецифікацій, де кожна модель ловить маніпуляції, які пропускає інша). Хмарні кредити конвертуються безпосередньо у знання:

| Навантаження | Що купують кредити |
|--------------|---------------------|
| **Масова обробка рішень ООПЗ** | Тисячі рішень органу оскарження (зібрано 4 400+, база росте), проаналізованих `oopz_researcher` через Gemini API → база прецедентів, що живить ризик-картки, профілі замовників, проекти рішень по скаргах і ризик-попередження конструктора ТД |
| **База знань вимог** | ~100 реальних ТД × 13 категорій закупівель у полегшеному extraction-режимі → базові пакети, медіани параметрів і безпечні діапазони конструктора ТД |
| **Бекфіл рішень АМКУ** | Full-text Gemini-аналіз рішень про змови на торгах → база доказів/логіки доказування для агента `investigation` |
| **Живі аналізи користувачів** | Виробничі аналізи тендерів/пропозицій для реальних користувачів — LLM-витрати працюючого продукту на кожен аналіз |
| **Хостинг і масштабування** | Перехід від одного Docker-хоста до керованої хмари: Cloud Run для автомасштабування агентів, Cloud SQL для баз PostgreSQL, Vertex AI як єдиний шлях доступу до моделей, Cloud Storage для документів і звітів, BigQuery для аналітики по учасниках закупівель |

Бази знань самопідсилюються: кожне додатково оброблене рішення покращує кожен наступний аналіз, а кожен звичайний прогін аналізу віддає вимоги і шаблони назад у бази як безкоштовний побічний продукт.

## Поточний стан (липень 2026)

Архітектура бекенда і повний пакет концепцій інтерфейсів (дизайн-система, кабінет учасника, конструктор замовника, ops-консоль, новини/статистика) спроектовані і взаємно звірені; ключові екрани супроводжуються статичними HTML-мокапами. Реалізація не починалась. Відкриті питання свідомо відкладені до implementation-спайків: технологія черги (Postgres-черга v1 чи Redis), вибір платіжного та e-mail провайдерів, стек повного фронтенду (за дизайном ізольований від бекенда OpenAPI-контрактом).
