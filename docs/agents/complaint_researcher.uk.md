# 📨 Complaint Researcher
**Статус:** Спроектовано (липень 2026), реалізація попереду — архітектура затверджена 07.07.2026
**Роль:** Досліджує скарги у сфері закупівель, подані до Органу оскарження (ООПЗ), і готує **проект рішення** для спеціаліста-людини

---

## Призначення

Коли учасник тендера подає скаргу до Органу оскарження публічних закупівель (ООПЗ), спеціаліст має зважити аргументи скаржника проти пояснень замовника, встановити факти, застосувати норми права і винести мотивоване рішення. `complaint_researcher` виконує це дослідження і готує **проект рішення** — спеціаліст-людина переглядає, править і вирішує.

**Проект, а не рішення — принципово.** Агент структурує, аргументує і пропонує; судить людина. Той самий принцип уже діє в аналізі техспецифікацій платформи («агент не судить, вирішує користувач»).

**Чому агент концептуально «майже готовий»:** три його постачальники вже існують. `tender_doc_researcher` дає патерни аналізу ТД, `bid_researcher` — патерни звірки пропозиції з вимогами, `oopz_researcher` — базу прецедентів. Нове — лише змагальний аналіз аргументів і генератор проекту рішення.

---

## Позиція в системі

```
tender_doc_researcher ──┐ (аналіз ТД, чеклісти)
bid_researcher ─────────┤ (звірка пропозицій з ТД)      → complaint_researcher
oopz_researcher ────────┘ (база прецедентів oopz_decisions)  → проект рішення ООПЗ

симетрія: amku_researcher → investigation (змови на торгах)
          oopz_researcher → complaint_researcher (скарги)
```

---

## Аргумент — одиниця розгляду

Скарга розкладається на окремі аргументи; кожен простежується через весь пайплайн:

```
argument:
  claim_verbatim        ← дослівна цитата доводу скаржника + джерело (документ/сторінка)
  claimed_violation     ← яку норму, на думку скаржника, порушено
  employer_position     ← verbatim відповідь замовника на ЦЕЙ довід, або явне
                          «замовник пояснень по цьому доводу не надав»
  type                  ← I / II / III (маршрутизація)
  referenced_documents  ← яких документів тендера стосується довід
```

Реальні рішення ООПЗ відповідають на кожен довід — агент зобов'язаний робити те саме. **Повнота контролюється детерміновано:** кожен витягнутий аргумент має завершитись вердиктом-пропозицією, інакше прогін не завершується (помилка, не попередження). Жоден аргумент не може бути мовчки відкинутий.

## Пайплайн

| Блок | Що відбувається |
|------|------------------|
| **А — Збір** | Скарга + додані матеріали + пояснення замовника (Prozorro API); контекст тендера (ТД, релевантні пропозиції, протоколи — лише те, чого стосується скарга); індексація документів і примусовий поділ мультидокументних файлів |
| **Б — Тріаж** | Перевірка приймальності (детерміновані правила ст.18 ЗУ №922: предмет, строки, статус скаржника, плата); класифікація типу скарги; екстракція аргументів (verbatim, Gemini) |
| **В — Аналіз** | Маршрутизація за типом (детермінована): тип I (умови ТД) → оскаржений пункт ТД проти норм, адаптація патернів `tender_doc_researcher`; тип II (відхилення / визначення переможця) → звірка пропозиції з ТД по оскарженим пунктам, адаптація патернів `bid_researcher` (включно з composite-принципом «інформація, а не файл»); підбір прецедентів з `oopz_decisions` |
| **Г — Синтез (Claude)** | Оцінка кожного аргументу → детермінована перевірка повноти → проект рішення у структурі реальних рішень ООПЗ |

Неприйнятна скарга — теж валідний вихід: проект «залишити без розгляду / припинити розгляд» з мотивацією.

## Змагальність за дизайном

Кожен аргумент аналізується не як «огляд документів», а як **зважування двох позицій**:

- Позиція скаржника — verbatim-цитата + заявлене порушення
- Позиція замовника — verbatim-цитата з пояснень; якщо відсутня, ця відсутність фіксується явно (сигнал спеціалісту, ніколи не пропускається мовчки)
- Встановлені факти (verbatim + джерела), застосовна норма (точна стаття/пункт) і 1–3 найближчі прецеденти з `oopz_decisions` з verbatim-висновками ООПЗ
- Пропозиція: **обґрунтований / необґрунтований / поза компетенцією** + рівень впевненості (низька впевненість чесно показується спеціалісту) + чернетка мотивувальної частини

Прецеденти є опорою мотивації — вони ніколи не замінюють аналіз. Аргументи без знайдених прецедентів позначаються явно («прецедентів у базі не знайдено») — це підвищує увагу спеціаліста, а не блокує прогін.

## Вихід

1. **Проект рішення (DOCX)** у структурі реальних рішень ООПЗ: вступна частина → обставини → позиції сторін по кожному доводу → мотивувальна частина по кожному доводу (факти → норма → оцінка, з прецедентними формулюваннями) → резолютивна частина. Резолютивна виводиться з вердиктів по аргументах детермінованими правилами агрегації.
2. **Аналітична записка спеціалісту:** таблиця аргумент → пропозиція → впевненість → прецеденти → на що звернути увагу (низька впевненість, відсутні пояснення замовника, немає прецедентів).

## Найчесніший eval у системі

Валідація виконується на **історичних скаргах, де рішення ООПЗ вже винесене**: проект агента порівнюється з фактичним рішенням, аргумент за аргументом, по резолютивному результату. Жодних синтетичних бенчмарків, жодного самооцінювання — природна ground truth з тисяч реальних розглядів.

---

## Скоуп і передумови

- **MVP покриває типи скарг I (умови ТД) та II (відхилення / визначення переможця)** — основну масу реальних скарг. Тип III (інші дії/бездіяльність) чесно позначається у проекті як поза скоупом.
- Без автоподання і без фінальних рішень — лише проект для спеціаліста.
- Прецеденти — виключно з наповненої таблиці `oopz_decisions`; агент не шукає рішення в інтернеті.
- **Критична передумова:** масове наповнення `oopz_decisions` агентом `oopz_researcher`. Без наповненої бази прецедентів агент був би «чернеткою без прецедентів» — тому ремонт і бекфіл каналу запису знань `oopz_researcher` передує реалізації цього агента.

## Поточний стан (липень 2026)

Архітектура та алгоритм розгляду повністю спроектовані 07.07.2026 (реконструйовано зі ст.18 ЗУ №922, Постанови КМУ №1178 і структури реальних рішень ООПЗ). Код реалізації ще не писався. Строки приймальності звірятимуться з актуальною редакцією закону на момент реалізації — норми воєнного часу неодноразово змінювались, тому жодні числа строків не захардкоджені у проект.
