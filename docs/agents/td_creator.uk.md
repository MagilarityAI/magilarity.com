# 🏗️ TD Creator
**Статус:** Спроектовано (липень 2026), реалізація попереду — архітектура затверджена 10.07.2026
**Роль:** Конструктор тендерної документації для замовників — збирає повний проект ТД з бази знань вимог з живими, прецедентно обґрунтованими ризик-попередженнями

---

## Призначення

Тендерний фахівець замовника — часто веде закупівлі «на пів ставки», без юриста — має два постійні болі: скарги, що зривають строки процедури, і невпевненість, чи законна його вимога. `td_creator` закриває обидва: замовник обирає категорію закупівлі, переглядає попередньо зібраний набір вимог (галочки + параметри) з попередженнями з реальної практики оскаржень і отримує **пропозицію** повної ТД у редагованому Word-файлі.

**Пропозиція, а не рішення:** фінальний документ і відповідальність за нього залишаються за уповноваженою особою замовника — той самий принцип «вирішує людина», що й у `complaint_researcher`.

**Унікальна позиція — ми володіємо обома сторонами ринку:** конструктор точно знає, як його ТД побачить наш власний аналізатор сторони учасника (`tender_doc_researcher`) і потенційний скаржник — *ще до* публікації.

```
td_creator (створення ТД, сторона замовника)  ← ЦЕЙ АГЕНТ
    ↓ публікація в Prozorro
tender_doc_researcher (аналіз ТД, сторона учасника) → bid_researcher
    ↓ скарги
complaint_researcher / oopz_researcher        investigation (змови)
```

## База знань вимог — головний актив

Вимоги видобуваються з **масової обробки реальних ТД** (через `tender_doc_researcher`) і кластеризуються по 13 канонічних категоріях закупівель:

| Шар | Визначення | Поведінка в UI |
|-----|------------|-----------------|
| Базові | зустрічаються у >50% ТД категорії | галочки ввімкнені |
| Поширені | 10–50% | вимкнені, додаються свідомо |
| Специфічні | <10% | вимкнені, додаються свідомо |

Практичний інсайт в основі дизайну: приблизно половина вимог у межах категорії однакова в різних замовників.

**Вимога = текст + параметри.** Оскаржують зазвичай не вимогу, а її параметр (3 сертифікати ISO замість 1; банківська гарантія 5% замість 3%). База зберігає медіанні значення і безпечний діапазон (p25–p75) по кожному параметру; конструктор підказує норму і одразу підсвічує вихід за неї.

Два канали наповнення тримають базу живою: полегшений extraction-режим (лише вимоги, без повного 16-пунктного аналізу — у рази дешевший за повний прогін `tender_doc_researcher`) для масового бекфілу, плюс hook, завдяки якому кожен звичайний прогін `tender_doc_researcher` віддає витягнуті вимоги як безкоштовний побічний продукт.

## Ризик-профілі з реальної практики ООПЗ (кілер-фіча)

Кожна канонічна вимога пов'язана з базою прецедентів `oopz_decisions`: скільки разів подібні вимоги оскаржувались, з якими результатами і в яких рішеннях. Попередження спрацьовують **наживо, в момент проставлення галочки чи введення параметра**:

- 🔴 «Подібна вимога оскаржувалась N разів, задоволено M (рішення №…). Безпечніше формулювання з практики: "…"»
- 🟡 «Параметр вище безпечного діапазону категорії (медіана X)»
- 🟡 Детерміновані анти-дискримінаційні патерни з практики (сертифікати на ім'я *учасника* замість виробника; вимоги локального офісу; комбінації вимог, типові для «заточення» під постачальника)

Попередження цитують номери рішень, а не абстрактну мораль. «Прецедентів не знайдено» чесно повідомляється як брак даних, а не нульовий ризик. Замовник може лишити ризиковану вимогу — його право і відповідальність — а кожне показане попередження фіксується у супровідній записці для уповноваженої особи.

## Нормативний каркас

Скелет ТД — офіційно затверджена **примірна форма ТД** плюс обов'язкові елементи ст.22 ЗУ №922 — вони рендеряться завжди, без галочок (галочки існують лише там, де в замовника є дискреція). Актуальна редакція примірної форми і закону отримується з бази законодавства на момент реалізації, ніколи не хардкодиться. Довідковий блок перелічує профільні НПА категорії (безпека харчових продуктів, ринок електроенергії, охоронна діяльність тощо), включно зі зв'язкою «цей НПА обґрунтовує такі типові вимоги».

---

## Два режими

| Режим | Флоу |
|-------|------|
| **(а) З нуля** | категорія → каркас + обов'язкові елементи ст.22 → базовий пакет з увімкненими галочками → специфічні вимоги + параметри з медіанами/безпечними діапазонами → генерація |
| **(б) Від чернетки замовника** | upload власної ТД → парсинг (reuse екстракції файлів `tender_doc_researcher`) → мапінг на базу вимог → «розпізнано X вимог; прогалини проти базового пакета категорії: […]; ризиковані місця: […]» → доповнення галочками |

---

## Детермінована збірка, мінімум LLM

Текст ТД **збирається детерміновано** з каркаса плюс канонічних формулювань бази з обраними параметрами — LLM не «пише ТД», тому нормативні формулювання мають нульовий ризик галюцинацій.

| Роль | Модель |
|------|--------|
| Екстракція вимог з ТД (наповнення бази), кластеризація формулювань | Gemini 2.5 Flash |
| Точкове згладжування специфічних текстових розділів | Claude Sonnet (точково) |
| Збірка ТД, параметри, експорт чеклиста, ризик-попередження | Без LLM — детерміновано |
| Self-check | пайплайн `tender_doc_researcher` |

## Вихід кожної генерації

- `ПРОПОЗИЦІЯ_ТД.docx` + додатки (форма пропозиції, технічні вимоги) — готово до правки у Word
- Супровідна записка (зроблений вибір, показані попередження, профільні НПА) для уповноваженої особи
- **Machine-readable чеклист вимог** у тому самому handoff-форматі, який `tender_doc_researcher` передає `bid_researcher` — тендери, народжені нашим конструктором, ніколи не потребують реверс-інжинірингу нашими ж аналізаторами
- Опційний **self-check**: згенерована ТД проганяється через повний 16-пунктний аналіз `tender_doc_researcher` — «як її побачить учасник і скаржник». Ціль: нуль порушень, нуль підстав для оскарження *до* публікації. Це водночас природний eval агента
- Опційний аналіз наданого замовником проєкту договору (reuse аналізатора договорів) — генерацію договорів агент свідомо **не** виконує
- При редагуванні вже опублікованої ТД: файл змін «було/стало» з нагадуваннями про строки (за правилами Постанови КМУ №1178)

## Архітектурний принцип: чистий core, тонкий UI

Весь функціонал (база вимог, ризик-аналіз, генерація DOCX) живе у чистій Python-бібліотеці з чіткими контрактами і нульовими залежностями від UI. Streamlit MVP — тонкий шар, що викликає цей core; майбутній повний фронтенд (через FastAPI) обгортає той самий core без переписування.

## Поточний стан (липень 2026)

Архітектура і обидва алгоритми (наповнення бази + майстер замовника) повністю спроектовані 10.07.2026. Код реалізації ще не писався. План наповнення бази: 2–3 пріоритетні категорії поступово до отримання хмарних кредитів; масовий прогін ~100 ТД × 13 категорій в extraction-режимі — після отримання кредитів. Поза скоупом MVP: генерація договорів, профіль конкуренції, мультилотовий майстер, Q&A-помічник.
