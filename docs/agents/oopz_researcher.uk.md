# ⚖️ OOPZ Researcher
**Статус:** Виробничий v2.0
**Роль:** Обробляє рішення ООПЗ (Органу оскарження публічних закупівель) і формує базу регуляторних прецедентів

---

## Призначення

Орган оскарження публічних закупівель (ООПЗ) видає рішення по скаргах у сфері закупівель. Ці рішення формують основний регуляторний прецедент щодо того, що є і що не є законним у тендерах.

oopz_researcher обробляє ці рішення і зберігає структурований аналіз у базі даних, яку tender_doc_researcher запитує при аналізі нових тендерів.

---

## Позиція в системі

```
oopz_researcher ← ЦЕЙ АГЕНТ
        ↓
    таблиця oopz_decisions (appdb)
        ↓ (зчитується)
tender_doc_researcher → використовує прецеденти при аналізі нових тендерів
```

## 10-кроковий аналіз кожного рішення

| Крок | Дія |
|------|-----|
| 1 | Створення циклу розслідування в investigation_cycles |
| 2 | Завантаження та валідація тексту рішення |
| 3 | LLM аналіз — витяг всіх фактів, позицій, обґрунтувань |
| 4 | Витяг метаданих з результату LLM |
| 5 | Оцінка якості — гібридна: 70% LLM + 30% валідатор |
| 6 | Юридична валідація — перевірка всіх посилань на норми по БД законодавства |
| 7 | Створення папки виводу (РІК/МІСЯЦЬ/ID_ЗАКУПІВЛІ) |
| 8 | Генерація звітів (DOCX + JSON + метадані) |
| 9 | Оновлення реєстрів Excel (2 файли з динамічними колонками) |
| 10 | Збереження в базу даних agent_memory |
| 10.5 | Запис у таблицю знань `appdb.oopz_decisions` (додано 07.07.2026) |

## 🛡️ Система захисту від галюцинацій — два фактори

**Фактор 1 (LLM аналіз):**
Моделі заборонено вигадувати факти. Кожне твердження має підтверджуватись прямою цитатою з тексту рішення.

**Фактор 2 (Юридична валідація):**
Всі посилання на нормативні акти перевіряються по повній базі українського законодавства (~286,634 нормативних документів в appdb). Посилання на неіснуючі статті позначаються прапором.

## 🗄️ Вивід в базу даних

Рішення зберігаються в agent_memory з повними структурованими даними і також індексуються в appdb.oopz_decisions для отримання іншими агентами:

```sql
-- Як tender_doc_researcher запитує ці дані:
SELECT decision_number, dk_code, complaint_type, 
       key_violation, analysis_json
FROM oopz_decisions
WHERE dk_code LIKE '45%'    -- Префікс ДК для будівництва
  AND is_analyzed = true
ORDER BY importance_score DESC
LIMIT 5
```

## 💎 Цінність для інших агентів

oopz_researcher є **фундаментом екосистеми MAGILARITY**:

| Агент | Як використовує дані ООПЗ |
|-------|--------------------------|
| `tender_doc_researcher` | Отримує релевантні прецеденти за кодом ДК для інформування аналізу |
| `bid_researcher` | Посилається на практику АМКУ щодо прийняття складових документів |
| `investigation` | Використовує патерни порушень для розслідування змов |

## 🔮 Стратегічна цінність

База рішень ООПЗ — це **унікальний актив** який неможливо швидко відтворити:

```
4,400+ структурованих рішень ООПЗ
        ↓
Патерни типових порушень замовників
        ↓
Статистика успішності різних типів скарг
        ↓
Прецедентна база для прогнозування
результатів оскарження
        ↓
Чим більше рішень — тим точніші висновки
агента tender_doc_researcher
```

Це **самонавчальна система** — кожне нове проаналізоване рішення ООПЗ підвищує якість аналізу всіх наступних тендерів.
