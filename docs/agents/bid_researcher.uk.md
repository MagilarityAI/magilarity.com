# 📋 Bid Researcher
**Статус:** Виробничий v3.4
**Роль:** Другий агент у pipeline — аналізує пропозиції учасників на відповідність вимогам тендеру

---

## Призначення

Отримує перелік документів від tender_doc_researcher та завантажує, індексує й аналізує кожен документ поданий кожним учасником тендеру. Формує юридичний звіт про відповідність для кожного учасника окремо.

---

## Позиція в системі

```
tender_doc_researcher → (чекліст через agent_handoffs) → bid_researcher ← ЦЕЙ АГЕНТ
                                                                 ↓
                                                    звіт відповідності на учасника
```

⚠️ **Критична залежність:** Не може працювати без чекліста від tender_doc_researcher. Якщо запис agent_handoffs відсутній — pipeline зупиняється з помилкою.

---

## Каскадна архітектура сит

```
checklist_receiver   → зчитує чекліст + winner_checklist + шлях до ТД документів
        ↓
bid_loader           → завантажує PDF/DOCX/.p7s/архіви з API ProZorro
                       класифікує файли за фазою: пропозиція / корекція_24год / переможець / підписання
        ↓
extract_archives     → розпаковує .zip/.rar/.7z; мапує архів→вміст
                       Пакети банківської гарантії → окрема папка з префіксом БГ001_/БС001_
        ↓
[КРОК 3б] office_converter → конвертує DOCX→PDF, XLSX→CSV (Gemini не приймає OOXML)
        ↓
[КРОК 4 / ЕТАП 1] stage1_indexer → 1 виклик Gemini на файл, усі фази → doc_index.json
        ↓
[КРОК 4в] correction_analyzer  → аналізує документи корекції 24год
[КРОК 4г] winner_doc_analyzer  → індексує документи фази переможця + чекліст переможця
[КРОК 4д] techspec_comparator  → ПОВНЕ порівняння технічних специфікацій
                       **АНСАМБЛЬ: Gemini 2.5 Flash + Gemini 3.5 Flash (дефолт, union знахідок)**;
                       gpt-5.1 (OpenAI, vision) — fallback через TECHSPEC_LLM=openai.
                       Специфікація замовника = еталон; специфікація учасника витягується
                       дослівно зі сканів (PyMuPDF → зображення)
[КРОК 4е] doc_compliance_verifier → ГЛИБОКА верифікація вмісту (Gemini 2.5 Flash)
                       Охоплення: банківська гарантія (УСІ фази) + кваліфікаційні критерії
                       ст.16 (досвід/обладнання/персонал/фінанси), керується прапором
                       requires_content_verification від tender_doc_researcher
        ↓
sieve1_simple        → 1 текстовий виклик → підтверджено / в обробці
        ↓
sieve2_tabular       → пакети по 8 PDF + маршрутизація по категоріях + аудит TechSpecComparator
        ↓
sieve3_scans         → пакети по 8 PDF + контекст сита 3
        ↓
arbitrator           → послідовно для кожного невирішеного документа → ✅ / ❌ / ⚠️
        ↓
final_report         → 1 виклик Gemini (синтез) → analysis.json
        ↓
_enrich_final_analysis → детермінований збагачення (без AI):
                       • технічні специфікації від TechSpecComparator (з ескалацією на перегляд)
                       • SAFEGUARD для банківських гарантій
                       • зв'язування архів↔вміст
        ↓
signature_scanner    → сканує .p7s файли → digital_signatures
                       Успадкування підпису: підписи архіву/Office
                       поширюються на вміст
        ↓
report_generator     → валідація → Node.js generate_docx.js → JSON + DOCX
        ↓
db_manager           → bid_documents + bid_analysis (UPSERT)
```

**Розділення на три моделі (свідома архітектура, не проміжна міграція):** основний каскад
(Етап 1, сита, арбітр, correction/winner/final_report) працює на легкій
`gemini-3.1-flash-lite-preview` заради швидкості та вартості. Порівняння технічних
специфікацій (крок 4д) — найкритичніша перевірка для тендерів товарів/обладнання — за
замовчуванням працює на **ансамблі 2.5+3.5 Flash**, ескалюючи до GPT-5.1 vision лише за
явним налаштуванням. Верифікація вмісту документів (крок 4е) працює на `gemini-2.5-flash`,
бо lite-модель надійно ОПИСУЄ файли, але ненадійно виявляє відсутні або невідповідні елементи
форми (паттерн echo/pattern-completion, підтверджено тестом на банківській гарантії).

## ⚖️ Ключовий правовий принцип — Інформація vs. Документ

Агент перевіряє **наявність інформації**, а не конкретного файлу:

| Тип збігу | Значення |
|-----------|----------|
| `direct` | Окремий виділений файл |
| `composite` | Інформація міститься в іншому документі |
| `no_match` | Інформація відсутня |

**Правова основа:** ст.16 ч.1, 3, 4 та ст.26 ч.1 Закону №922-VIII; п.43 Постанови КМУ №1178; прецедент АМКУ — замовник не може відхилити якщо інформація міститься у складовому документі.

## 📅 Фази документів

Кожен завантажений файл класифікується за датою публікації відносно строків тендеру:

| Фаза | Діапазон дат | Папка |
|------|-------------|-------|
| `proposal` | ≤ tenderPeriod.endDate | документи_пропозиції/ |
| `correction_24h` | endDate < дата ≤ awards[0].date | документи_24_години/ |
| `winner` | дата > awards[0].date | документи_переможця_4_дні/ |
| `signing` | після строку оскарження | документи_підписання/ |

## 🔍 Порівняння технічних специфікацій (Vision LLM)

Для технічних товарів і будівництва:

```
PDF сторінки → PNG зображення (PyMuPDF)
Специфікація замовника з ТД = еталон
Специфікація учасника витягується дослівно зі сканованих документів
Виявляє: відхилення, додані пункти, пропуски
Результат: technical_specs_compliance[] з критичністю (critical / substantial)
```

---

## 🔎 Верифікація вмісту документів (Analysis LLM)

Глибока верифікація документів де наявності недостатньо:

**Завжди перевіряється:**
- Банківська гарантія (відповідність формі згідно додатку тендеру)
- Кваліфікаційні критерії ст.16 (досвід/аналогічні договори, обладнання, персонал, фінансова спроможність — перевірка порогів)
- Технічна специфікація

Активується прапором `requires_content_verification: true` встановленим tender_doc_researcher для кожної позиції чекліста.

## 📊 Шкала вердиктів

| Вердикт | Значення | Рекомендована дія |
|---------|----------|------------------|
| `compliant` | Всі вимоги виконані | Прийняти до оцінки |
| `requires_correction` | Виправні недоліки | Надати 24год (п.43 КМУ 1178) |
| `non_compliant` | Критичні недоліки | Відхилити (п.44 КМУ 1178) |
| `requires_user_review` | Невирішуване або автовиправлене | Ручна перевірка |
| `mixed` | Є і виправні і критичні | Потрібен детальний аналіз |

## 🛡️ Автоматичні захисні механізми (без AI — детерміновані)

**SAFEGUARD Банківська гарантія:** Якщо обраний порожній placeholder замість реальної БГ → позиція підвищується до `requires_user_review`. Запобігає хибному відхиленню.

**Ескалація TechSpec:** Якщо порівняння технічних специфікацій або аудит виявив відхилення → позиція `compliant` підвищується до `requires_user_review` (прапор: `techspec_escalated`). Тільки вгору (суворіший статус не пом'якшується).

---

## ✍️ Аналіз цифрових підписів

- Парсинг .p7s файлів (PKCS#7 через asn1crypto)
- Виявляє ім'я підписанта, організацію, тип ключа (КЕП/УЕП)
- **Успадкування підпису:** підпис архіву поширюється на весь витягнутий вміст; оригінальний підпис Office поширюється на конвертований PDF (поле: `signed_via`)

## 🔄 Інкрементальний аналіз

Розроблений для поетапної публікації документів у ProZorro:

```
Прогін 1: документи пропозиції → Звіт v1
Прогін 2: додані документи корекції 24год → Звіт v2
Прогін 3: додані документи переможця → Звіт v3
```

Файли вже на диску не перезавантажуються. Нові файли Етапу 1 автоматично інвалідують контрольні точки сит.

## 🏷️ Коди статусів позицій чекліста

| Код | Значення |
|-----|----------|
| **В** | Відповідає — документ присутній і відповідає вимогам |
| **24** | Корекція 24год — присутній але виправний недолік |
| **В!** | Відхилення — критична невідповідність |
| **УК** | Умовно — потребує верифікації користувача |
| **ВД** | Відсутній — документ або інформація не подані |

Автоматичні детерміновані ескалації до `УК` (лише вгору, суворіший статус не пом'якшується):
**SAFEGUARD** — обрано порожній placeholder банківської гарантії замість реальної
high-confidence знахідки; **ескалація TechSpec** — повне порівняння або аудит сита 2 виявили
відхилення/прогалини.
